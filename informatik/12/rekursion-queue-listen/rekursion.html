Kapitel 1

Arrays und ihre Probleme

📊 Was sind Arrays?

Definition: Arrays (Felder) dienen der Speicherung einer festen Anzahl von Objekten gleichen Datentyps, welche über einen Index (Positionszahl) adressiert werden können.

Interaktive Array-Visualisierung

Aktuell belegt: 0 / 4

⚠️ Probleme mit Arrays

Vervollständige den Lückentext mit den passenden Wörtern:

Die Belegung des Arrays ist recht umständlich. Das Feld muss zunächst erstellt und initialisiert werden, um anschließend die einzelnen Elemente an die richtige Stelle setzen zu können.

Auch kann das Array nicht einfach erweitert werden, sondern es muss ein neues Array erstellt und alle Elemente kopiert werden.

Zusätzlich verbraucht das Array sehr viel Speicher, auch wenn die Objekte noch nicht initialisiert sind, da bereits zum Zeitpunkt der Erstellung des Arrays entsprechend RAM allokiert wird.

🎯 Aufgabe 1: Array-Grundlagen

  1. Erstelle ein Array mit 10 Plätzen für Strings
  2. Belege 5 zufällige Positionen mit Namen
  3. Durchlaufe das Array und fülle alle leeren Felder mit "Leer"
  4. Gib alle Elemente mit ihrem Index aus

💡 Hilfestellung

Array erstellen:

String[] namen = new String[10];

Zufällige Position:

int position = (int)(Math.random() * 10);

Durchlaufen:

for (int i = 0; i < namen.length; i++) {
    if (namen[i] == null) {
        namen[i] = "Leer";
    }
}

💻 Programmierbereich - Array Übung

✅ Vorteile von Arrays

  • Schneller Zugriff über Index (O(1))
  • Einfache Implementierung
  • Geringer Overhead
  • Zusammenhängender Speicher

❌ Nachteile von Arrays

  • Feste Größe - nicht erweiterbar
  • Speicherverschwendung bei Unterauslastung
  • Einfügen/Löschen aufwendig
  • Muss vorab dimensioniert werden

🔍 Die Lösung: Dynamische Datenstrukturen

Um diese Probleme zu beheben, gibt es eine Reihe von Datenstrukturen. In Java sind diese als Subinterfaces von java.util.Collection zusammengefasst.

Im Folgenden werden wir die Datenstruktur Queue (Warteschlange) von Grund auf selbst implementieren!

Kapitel 2

Die Queue - Konzept und Idee

🔄 Was ist eine Queue?

Queue (Warteschlange): Eine Datenstruktur, die Elemente nach dem FIFO-Prinzip verwaltet.

FIFO = First In, First Out - Das erste Element, das hinzugefügt wird, wird auch als erstes wieder entnommen.

Interaktive Queue-Visualisierung

Anzahl Elemente: 0

💡 Die grundlegende Idee

Statt ein großes Array zu verwenden, bei dem viele Plätze leer bleiben können, nutzen wir eine verkettete Struktur:

Jedes Element kennt:

  • Seine eigenen Daten (z.B. Patient, Taxi, Kunde)
  • Einen Verweis auf seinen Nachfolger

Die Queue selbst muss nur zwei Referenzen speichern:

  • anfang - Verweis auf das erste Element
  • ende - Verweis auf das letzte Element

🎯 Queue Operationen

add(element) - Hinzufügen

Fügt ein Element am Ende der Queue hinzu.

  • Neues Element wird erstellt
  • Aktuelles Ende verweist auf neues Element
  • Neues Element wird zum neuen Ende

poll() - Entnehmen

Entfernt und gibt das Element am Anfang zurück.

  • Erstes Element wird gespeichert
  • Anfang wird auf Nachfolger gesetzt
  • Gespeichertes Element wird zurückgegeben

peek() - Ansehen

Gibt das erste Element zurück, ohne es zu entfernen.

  • Anfang wird zurückgegeben
  • Queue bleibt unverändert

✨ Vorteile der Queue gegenüber Arrays

🚀
Dynamische Größe

Die Queue kann beliebig wachsen - keine feste Größe nötig!

💾
Effizienter Speicher

Es wird nur Speicher für tatsächlich vorhandene Elemente belegt.

⚡
Schnelle Operationen

Hinzufügen und Entfernen sind O(1) - konstante Zeit!

🎯
Klare Struktur

FIFO-Prinzip wird automatisch eingehalten.

📚 Merke

Die Beziehung zwischen Objekten der gleichen Klasse nennt man rekursive Beziehung. Die „hat Nachfolger"-Beziehung zwischen den einzelnen Elementen stellt eine solche dar.

Zur Umsetzung in Programmcode benötigen wir ein Referenzattribut mit gleichem Datentyp wie die Klasse selbst.

Kapitel 3

Queue bauen - Teil 1: Die Grundstruktur

🎯 Aufgabe: Erstelle die Klasse ListElement

Wir beginnen mit dem Baustein unserer Queue - dem einzelnen Element in der Kette.

  1. Erstelle eine Klasse ListElement
  2. Füge ein Attribut daten vom Typ Patient hinzu
  3. Füge ein Attribut nachfolger vom Typ ListElement hinzu (rekursive Beziehung!)
  4. Erstelle einen Konstruktor, der die Daten entgegennimmt
  5. Erstelle Getter und Setter für beide Attribute

💡 Hilfestellung - Struktur

public class ListElement {
    // Attribute
    private Patient daten;
    private ListElement nachfolger;  // ← Rekursive Beziehung!
    
    // Konstruktor
    public ListElement(Patient daten) {
        this.daten = daten;
        // nachfolger bleibt erstmal null
    }
    
    // Getter und Setter
    public Patient getDaten() {
        return daten;
    }
    
    public ListElement getNachfolger() {
        return nachfolger;
    }
    
    public void setNachfolger(ListElement nachfolger) {
        this.nachfolger = nachfolger;
    }
}

🔍 Verstehe die rekursive Beziehung

Das Besondere: Ein ListElement hat ein Attribut vom Typ ListElement!

Dies ermöglicht die Verkettung: Element 1 → Element 2 → Element 3 → ...

Wichtig: Der nachfolger ist zunächst null. Das bedeutet "kein Nachfolger" bzw. "Ende der Kette".

💻 Programmierbereich - ListElement

🎓 Verständnisfrage

Warum ist es wichtig, dass nachfolger zunächst null ist?

✅ Antwort

null signalisiert das Ende der Kette. Wenn wir durch die Queue iterieren, wissen wir: "Wenn nachfolger == null, dann sind wir am Ende angekommen."

Ohne diese Konvention würden wir nicht wissen, wann wir aufhören sollen!

Kapitel 4

Queue bauen - Teil 2: Die Queue-Klasse

🎯 Aufgabe: Erstelle die Queue-Klasse

Jetzt bauen wir die Verwaltungsklasse, die die Kette von ListElementen organisiert.

  1. Erstelle eine Klasse PatientQueue
  2. Füge Attribute anfang und ende vom Typ ListElement hinzu
  3. Erstelle einen Konstruktor (anfang und ende sind zunächst null)
  4. Implementiere die Methode add(Patient patient)

💡 Hilfestellung - add() Methode

Die add-Methode muss zwei Fälle unterscheiden:

Fall 1: Queue ist leer (anfang == null)

if (anfang == null) {
    ListElement neuesElement = new ListElement(patient);
    anfang = neuesElement;
    ende = neuesElement;
    return true;
}

Fall 2: Queue hat bereits Elemente

else {
    ListElement neuesElement = new ListElement(patient);
    ende.setNachfolger(neuesElement);  // Altes Ende verweist auf neues
    ende = neuesElement;                // Neues Element wird zum Ende
    return true;
}

🔍 Warum zwei Referenzen?

anfang: Ermöglicht schnelles Entnehmen (poll) - O(1)

ende: Ermöglicht schnelles Hinzufügen (add) - O(1)

Ohne die ende-Referenz müssten wir bei jedem add() die gesamte Kette durchlaufen, um das Ende zu finden - das wäre O(n)!

Kapitel 5

Queue bauen - Teil 3: poll() und peek()

🎯 Aufgabe: Implementiere poll() und peek()

Jetzt fehlen noch die Methoden zum Entnehmen und Ansehen des ersten Elements.

  1. Implementiere poll() - Entfernt und gibt das erste Element zurück
  2. Implementiere peek() - Gibt das erste Element zurück, ohne es zu entfernen
  3. Teste beide Methoden ausführlich

💡 Hilfestellung - poll() Methode

Schritt-für-Schritt Anleitung:

  1. Prüfe, ob Queue leer ist (anfang == null) → return null
  2. Speichere das erste Element zwischen
  3. Setze anfang auf den Nachfolger des ersten Elements
  4. Wenn anfang jetzt null ist, setze auch ende auf null (Queue ist leer)
  5. Gib das gespeicherte Element zurück
public ListElement poll() {
    if (anfang == null) {
        return null;  // Queue ist leer
    }
    
    ListElement entferntes = anfang;
    anfang = anfang.getNachfolger();
    
    if (anfang == null) {
        ende = null;  // Queue ist jetzt leer
    }
    
    return entferntes;
}

peek() ist einfacher:

public ListElement peek() {
    return anfang;  // Einfach anfang zurückgeben
}

⚠️ Wichtige Randfälle

Leere Queue: Was passiert, wenn poll() auf eine leere Queue aufgerufen wird?

→ Wir geben null zurück und geben ggf. eine Warnung aus.

Letztes Element: Was passiert, wenn das letzte Element entfernt wird?

→ Sowohl anfang als auch ende müssen auf null gesetzt werden!

💻 Programmierbereich - poll() und peek()

📊 Visualisierung: poll() in Aktion

Ausgangssituation:

anfang → [Hans] → [Anna] → [Peter] → null
ende → [Peter]

Nach poll():

anfang → [Anna] → [Peter] → null
ende → [Peter]
Rückgabe: [Hans]

Nach erneutem poll():

anfang → [Peter] → null
ende → [Peter]
Rückgabe: [Anna]

Nach letztem poll():

anfang → null
ende → null
Rückgabe: [Peter]

🏆 Herausforderung: Erweitere die Queue!

Implementiere zusätzliche Methoden:

  • size() - Gibt die Anzahl der Elemente zurück
  • isEmpty() - Prüft, ob Queue leer ist
  • clear() - Leert die Queue komplett

✅ Lösungsvorschlag

public int size() {
    int count = 0;
    ListElement current = anfang;
    while (current != null) {
        count++;
        current = current.getNachfolger();
    }
    return count;
}

public boolean isEmpty() {
    return anfang == null;
}

public void clear() {
    anfang = null;
    ende = null;
}

Kapitel 6

Interfaces - Flexibilität durch Abstraktion

🔌 Was sind Interfaces?

Interface: Ein Vertrag, der festlegt, welche Methoden eine Klasse bereitstellen muss, ohne die Implementierung vorzugeben.

Vergleich: Ein Interface ist wie ein EU-Gesetz:

  • Es legt fest, was getan werden muss
  • Es legt nicht fest, wie es getan wird
  • Jedes Land (Klasse) kann es unterschiedlich umsetzen
  • Aber alle müssen die Mindestanforderungen erfüllen

🚗 Beispiel: Auto-Interface

Nahezu alle Autos haben die gleichen grundsätzlichen Steuerelemente (Lenkrad, Gaspedal, Bremse), die sie zur Verfügung stellen müssen.

Die Modelle und deren genaues Verhalten unterscheiden sich jedoch je nach Leistung oder ob sie über ABS und Servolenkung verfügen.

interface Auto {
    void lenken(int grad);
    void beschleunigen();
    void bremsen();
}

class SportWagen implements Auto {
    // Schnelle Beschleunigung
}

class Familienauto implements Auto {
    // Sanfte Beschleunigung
}

📋 Eigenschaften von Interfaces

Attribute

  • Alle Attribute sind implizit public static final
  • Also: öffentliche Konstanten
  • Beispiel: int MAX_SPEED = 200;

Methoden

  • Nur Methodenköpfe (Signaturen)
  • Kein Methodenrumpf
  • Implizit public
  • Müssen implementiert werden

Implementierung

  • Klasse kann mehrere Interfaces implementieren
  • Keyword: implements
  • Alle Methoden müssen überschrieben werden

🎯 Aufgabe: DatenElement Interface

Erstelle ein Interface, das verschiedene Datentypen in unserer Queue ermöglicht.

  1. Erstelle ein Interface DatenElement
  2. Definiere eine Methode String toString()
  3. Lasse Patient und Taxi das Interface implementieren
  4. Ändere ListElement so, dass es DatenElement statt Patient verwendet

💡 Hilfestellung

Interface erstellen:

public interface DatenElement {
    String toString();
}

Patient implementiert Interface:

public class Patient implements DatenElement {
    // ... Attribute ...
    
    @Override
    public String toString() {
        return "Patient: " + name + " (" + alter + " Jahre)";
    }
}

ListElement verwendet Interface:

public class ListElement {
    private DatenElement daten;  // Statt Patient!
    private ListElement nachfolger;
    
    public ListElement(DatenElement daten) {
        this.daten = daten;
    }
    // ...
}

💻 Programmierbereich - Interfaces

✨ Vorteile von Interfaces

🔄
Flexibilität

Die Queue kann jetzt beliebige Datentypen aufnehmen - nicht nur Patienten!

🔗
Lose Kopplung

Die Queue ist unabhängig von konkreten Klassen - sie kennt nur das Interface.

🎯
Hohe Kohäsion

Jede Klasse hat eine klare, wohldefinierte Aufgabe.

📦
Wiederverwendbarkeit

Die gleiche Queue-Implementierung funktioniert für alle Datentypen!

📚 Wichtige Konzepte

Kohäsion: Wie gut eine Programmeinheit eine logische Aufgabe abbildet. Starke Kohäsion ist gut!

Kopplung: Wie stark verschiedene Module voneinander abhängen. Lose Kopplung ist gut!

Kapitel 7

Listen - Mehr als nur eine Queue

📝 Was ist eine Liste?

Liste: Eine Datenstruktur mit dem gleichen Grundaufbau wie eine Queue, aber mit erweiterten Funktionen.

Gemeinsamkeiten mit Queue:

  • Verkettete Struktur (Knoten mit Nachfolgern)
  • Dynamische Größe
  • Referenz auf den Anfang

Unterschiede zur Queue:

  • Einfügen an beliebiger Stelle möglich
  • Entfernen an beliebiger Stelle möglich
  • Zugriff auf beliebige Elemente
  • Nur eine Referenz nötig (anfang)

🔄 Queue (Warteschlange)

Operationen:
  • add() - Am Ende hinzufügen
  • poll() - Am Anfang entnehmen
  • peek() - Anfang ansehen
Prinzip:

FIFO - First In, First Out

Referenzen:

anfang + ende

📝 Liste

Operationen:
  • addFirst() - Am Anfang einfügen
  • addLast() - Am Ende einfügen
  • addBefore() - Vor Element einfügen
  • remove() - Beliebiges Element entfernen
  • contains() - Element suchen
  • size() - Länge berechnen
Prinzip:

Flexibler Zugriff

Referenzen:

nur anfang

🎯 Anwendungsfälle

Queue verwenden für:

  • Warteschlangen (Drucker, Tickets, etc.)
  • Aufgaben-Warteschlangen (Task Queues)
  • Breadth-First-Search Algorithmen
  • Wenn strikte FIFO-Reihenfolge wichtig ist

Liste verwenden für:

  • Flexible Datensammlungen
  • Wenn Einfügen/Löschen an beliebiger Stelle nötig ist
  • Wenn Suche nach bestimmten Elementen wichtig ist
  • Sortierte Datenstrukturen

🔧 Neue Herausforderungen bei Listen

Rekursion

Viele Listenoperationen nutzen Rekursion - eine Methode ruft sich selbst auf.

Beispiel: Um das Ende zu finden, fragt jeder Knoten seinen Nachfolger: "Bist du das Ende?"

Durchreichen

Operationen werden von Knoten zu Knoten durchgereicht, bis der richtige Knoten gefunden ist.

Beispiel: "Füge vor Element X ein" → Jeder Knoten prüft: "Bin ich X?" und gibt sonst weiter.

Randfälle

Besondere Aufmerksamkeit für:

  • Leere Liste
  • Erstes Element
  • Letztes Element
  • Liste mit nur einem Element

Kapitel 8

Liste implementieren

🎯 Aufgabe: Erstelle die Klassen MeineListe und Knoten

Wir bauen eine Liste von Grund auf - ähnlich wie die Queue, aber mit mehr Funktionen.

  1. Erstelle die Klasse Knoten (ähnlich wie ListElement)
  2. Erstelle die Klasse MeineListe (nur mit Referenz auf anfang)
  3. Implementiere addLast() - fügt am Ende hinzu
  4. Implementiere addFirst() - fügt am Anfang hinzu

💡 Hilfestellung - Grundstruktur

Knoten Klasse:

public class Knoten {
    private DatenElement daten;
    private Knoten nachfolger;
    
    public Knoten(DatenElement daten) {
        this.daten = daten;
    }
    
    // Getter und Setter
    // ...
}

MeineListe Klasse:

public class MeineListe {
    private Knoten anfang;
    
    public MeineListe() {
        // anfang ist automatisch null
    }
    
    // Methoden folgen...
}

addFirst() - Einfach!

public void addFirst(DatenElement daten) {
    Knoten neuerKnoten = new Knoten(daten);
    neuerKnoten.setNachfolger(anfang);
    anfang = neuerKnoten;
}

🤔 Herausforderung: addLast() ohne ende-Referenz

Bei der Queue hatten wir eine ende-Referenz. Bei der Liste nicht!

Problem: Wie finden wir das Ende der Liste?

Lösung: Rekursion! Jeder Knoten fragt seinen Nachfolger.

Idee:

  1. Liste ruft anfang.add(daten) auf
  2. Knoten prüft: Habe ich einen Nachfolger?
  3. Wenn ja: Gib die Aufgabe an Nachfolger weiter (Rekursion!)
  4. Wenn nein: Ich bin das Ende, ich füge das Element an

💻 Programmierbereich - Liste Grundfunktionen

📊 Visualisierung: Rekursiver Aufruf bei addLast()

Ausgangssituation: Liste mit 3 Elementen

anfang → [Taxi 101] → [Taxi 102] → [Taxi 103] → null

Aufruf: liste.addLast(new Taxi(104, "Neu"))

Ablauf:

  1. Liste: "anfang, füge das Element hinzu!"
  2. Knoten 101: "Ich habe einen Nachfolger, ich gebe weiter..."
  3. Knoten 102: "Ich habe einen Nachfolger, ich gebe weiter..."
  4. Knoten 103: "Ich habe keinen Nachfolger! Ich füge an: [Taxi 104]"

Ergebnis:

anfang → [Taxi 101] → [Taxi 102] → [Taxi 103] → [Taxi 104] → null

🏆 Erweiterte Aufgabe: Weitere Methoden

Implementiere folgende Methoden:

  • length() - Gibt die Anzahl der Elemente zurück (rekursiv!)
  • contains(DatenElement daten) - Prüft, ob Element in Liste ist (rekursiv!)
  • remove(DatenElement daten) - Entfernt ein Element (rekursiv!)

✅ Lösungsansatz

length() in MeineListe:

public int length() {
    if (anfang == null) {
        return 0;
    }
    return anfang.length();
}

length() in Knoten:

public int length() {
    if (nachfolger == null) {
        return 1;  // Ich bin der letzte
    }
    return 1 + nachfolger.length();  // 1 + Rest
}

Tipp für remove(): Der Knoten muss seinen Nachfolger "überspringen" können!

Kapitel 9

Rekursion verstehen und meistern

♻️ Was ist Rekursion?

Rekursion: Eine Methode ruft sich selbst auf.

Dies führt zu einer Verschachtelung von Methodenaufrufen, die theoretisch unendlich weiterlaufen könnte.

Wichtig: Es muss eine Abbruchbedingung geben!

Die Abbruchbedingung greift, wenn:

  • Die Aufgabe erfolgreich gelöst werden kann (z.B. Ende der Liste erreicht)
  • Die Bedingung nicht mehr erfüllt ist (z.B. Element nicht gefunden)

🎯 Beispiel: Fakultät berechnen

Mathematische Definition:

n! = n × (n-1)!

5! = 5 × 4! = 5 × 4 × 3! = 5 × 4 × 3 × 2 × 1 = 120

Rekursive Implementierung:

public int fakultaet(int n) {
    if (n <= 1) {
        return 1;  // Abbruchbedingung!
    }
    return n * fakultaet(n - 1);  // Rekursiver Aufruf
}

Ablauf für fakultaet(5):

  1. fakultaet(5) = 5 × fakultaet(4)
  2. fakultaet(4) = 4 × fakultaet(3)
  3. fakultaet(3) = 3 × fakultaet(2)
  4. fakultaet(2) = 2 × fakultaet(1)
  5. fakultaet(1) = 1 ← Abbruch!
  6. Rückwärts auflösen: 2×1 = 2, 3×2 = 6, 4×6 = 24, 5×24 = 120

🎯 Rekursion in Datenstrukturen

add() - Hinzufügen

Idee: "Wenn ich nicht das Ende bin, gib die Aufgabe weiter."

if (nachfolger == null) {
    // Abbruch: Ich bin das Ende
    nachfolger = new Knoten(daten);
} else {
    // Rekursion: Weitergeben
    nachfolger.add(daten);
}

length() - Länge berechnen

Idee: "Ich zähle 1, plus die Länge des Rests."

if (nachfolger == null) {
    // Abbruch: Ich bin der letzte
    return 1;
} else {
    // Rekursion: 1 + Rest
    return 1 + nachfolger.length();
}

contains() - Suchen

Idee: "Bin ich es? Wenn nicht, frag den Nachfolger."

if (daten.equals(gesucht)) {
    // Abbruch: Gefunden!
    return true;
} else if (nachfolger == null) {
    // Abbruch: Ende, nicht gefunden
    return false;
} else {
    // Rekursion: Weiterfragen
    return nachfolger.contains(gesucht);
}

🎯 Aufgabe: Implementiere remove() rekursiv

Die schwierigste Methode: Ein Element aus der Liste entfernen.

Herausforderung: Der Knoten muss seinen Nachfolger "überspringen" können!

  1. In MeineListe: Prüfe, ob anfang das gesuchte Element ist
  2. Wenn ja: Setze anfang auf anfang.getNachfolger()
  3. Wenn nein: Gib die Aufgabe an anfang weiter (rekursiv)
  4. In Knoten: Prüfe, ob mein Nachfolger das gesuchte Element ist
  5. Wenn ja: Überspringe ihn (setze nachfolger auf nachfolger.getNachfolger())
  6. Wenn nein: Gib die Aufgabe an Nachfolger weiter

💡 Hilfestellung - remove()

In MeineListe:

public void remove(DatenElement daten) {
    if (anfang == null) {
        return;  // Liste ist leer
    }
    
    if (anfang.getDaten().equals(daten)) {
        // Erstes Element soll entfernt werden
        anfang = anfang.getNachfolger();
    } else {
        // Gib die Aufgabe an anfang weiter
        anfang.remove(daten);
    }
}

In Knoten:

public void remove(DatenElement daten) {
    if (nachfolger == null) {
        return;  // Kein Nachfolger, nichts zu tun
    }
    
    if (nachfolger.getDaten().equals(daten)) {
        // Mein Nachfolger soll entfernt werden
        nachfolger = nachfolger.getNachfolger();
    } else {
        // Gib die Aufgabe weiter
        nachfolger.remove(daten);
    }
}

Kapitel 10

Das Kompositum Pattern

🎭 Was ist das Kompositum Pattern?

Kompositum (Composite Pattern): Ein Entwurfsmuster, das verwendet wird, um Fallunterscheidungen zu vermeiden.

Alle Gemeinsamkeiten werden in einer abstrakten Klasse zusammengefasst. Für jede Ausprägung gibt es eine eigene Klasse mit eigenem Verhalten.

Ziel: Vermeidung von if (nachfolger == null) Abfragen!

Lösung: Statt null verwenden wir ein Abschluss-Objekt.

🔍 Problem: Viele null-Checks

Bisheriger Code:

public Knoten add(DatenElement daten) {
    if (nachfolger == null) {  // ← null-Check
        nachfolger = new Knoten(daten);
        return this;
    } else {
        nachfolger = nachfolger.add(daten);
        return this;
    }
}

public int length() {
    if (nachfolger == null) {  // ← null-Check
        return 1;
    } else {
        return 1 + nachfolger.length();
    }
}

Problem: In jeder Methode müssen wir prüfen, ob nachfolger == null ist!

🎯 Die Lösung: Drei Klassen statt zwei

ListenElement (abstrakt)

Definiert alle Methoden, die sowohl Knoten als auch Abschluss haben müssen.

  • add()
  • remove()
  • length()
  • contains()

Knoten (konkret)

Ein normales Element in der Liste mit Daten und Nachfolger.

  • Hat Daten
  • Hat Nachfolger (ListenElement!)
  • Implementiert alle Methoden

Abschluss (konkret)

Markiert das Ende der Liste - ersetzt null!

  • Keine Daten
  • Kein Nachfolger
  • Einfache Implementierungen

🔑 Abstrakte Klassen vs. Interfaces

Abstrakte Klasse:

  • Kann nicht instanziiert werden (kein new)
  • Kann abstrakte UND konkrete Methoden haben
  • Kann Attribute haben
  • Hat einen Konstruktor (für Subklassen)

Interface:

  • Nur Methodensignaturen (Stubs)
  • Keine Implementierungen (außer default)
  • Nur Konstanten als Attribute
  • Kein Konstruktor

💻 Programmierbereich - Kompositum Pattern

📊 Vorher vs. Nachher

❌ Ohne Kompositum

public int length() {
    if (nachfolger == null) {
        return 1;
    } else {
        return 1 + nachfolger.length();
    }
}

Viele if-Abfragen nötig!

✅ Mit Kompositum

// In Knoten:
public int length() {
    return 1 + nachfolger.length();
}

// In Abschluss:
public int length() {
    return 0;
}

Keine if-Abfragen mehr!

✨ Vorteile des Kompositum Patterns

🎯
Keine null-Checks

Der Code wird einfacher und weniger fehleranfällig.

🔄
Einheitliche Behandlung

Knoten und Abschluss werden gleich behandelt - beide sind ListenElemente.

📦
Klare Verantwortlichkeiten

Jede Klasse weiß genau, was sie tun muss.

🚀
Leichte Erweiterbarkeit

Neue Methoden können einfach hinzugefügt werden.

🏆 Verständnisfrage

Warum ist es wichtig, dass eine leere Liste einen Abschluss als anfang hat und nicht null?

✅ Antwort

Wenn anfang ein Abschluss ist, können wir direkt anfang.add(), anfang.length() etc. aufrufen, ohne vorher zu prüfen, ob anfang == null ist!

Der Abschluss "weiß", wie er sich verhalten muss (z.B. length() gibt 0 zurück, add() erstellt einen neuen Knoten).

Das ist das Herzstück des Kompositum Patterns: Polymorphismus statt Fallunterscheidungen!

📖 Wichtige Begriffe

FIFO (First In, First Out)

Prinzip der Queue: Das erste Element, das hinzugefügt wird, wird auch als erstes entnommen.

Rekursion

Eine Methode ruft sich selbst auf, um ein Problem schrittweise zu lösen.

Kohäsion

Maß dafür, wie gut eine Klasse eine einzelne, wohldefinierte Aufgabe erfüllt.

Kopplung

Maß für die Abhängigkeit zwischen verschiedenen Klassen. Lose Kopplung ist erstrebenswert.

Polymorphismus

Verschiedene Klassen können die gleiche Schnittstelle unterschiedlich implementieren.

Abstraktion

Trennung von Schnittstelle (was) und Implementierung (wie).
